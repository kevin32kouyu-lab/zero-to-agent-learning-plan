## 🚀 第1天：Coze（字节）低代码平台体验

### 上午（2h）：平台入门

**什么是Coze？**
- 字节跳动出品的低代码Agent开发平台
- 国内直接访问，不用科学上网
- 免费额度够用，适合快速原型
- 预置大量插件（搜索、代码执行、图片生成等）

**第一步：注册登录**
- 官网：https://www.coze.cn/
- 用手机号直接注册，开箱即用

**核心概念理解：**
| 概念 | 作用 |
|------|------|
| **Bot** | 就是你的Agent，一个Bot对应一个智能体 |
| **插件** | 工具集合，比如联网搜索、调用API |
| **知识库** | 上传文档，Coze帮你做RAG |
| **工作流** | 可视化编排Agent流程，不用写代码 |
| **发布** | 一键发布到Telegram、Discord、网页等 |

---

### 下午（2h）：动手做一个知识库问答Bot

**跟着这个步骤做，30分钟就能做好：**

1. **创建Bot**
   - 点击"创建Bot"，起个名字，比如"我的学习笔记"
   - 填写角色描述："你是一个助手，基于我的知识库回答问题，如果找不到就说不知道"

2. **上传文档到知识库**
   - 左侧"知识库" → 创建知识库 → 上传你的PDF/Word/TXT
   - Coze自动分块、生成Embedding、建立索引
   - 几分钟搞定，不用写一行代码

3. **Bot关联知识库**
   - Bot编辑页 → 找到"知识库" → 勾选你刚创建的
   - 配置检索参数（默认一般就够用）

4. **测试对话**
   - 右侧测试窗口直接聊，问一个文档里的问题
   - 看回答对不对，检查检索准不准

**完成了！** 从0到做好一个可用知识库Agent，不到2小时，这就是低代码的效率。

## 🔀 第2天：n8n工作流自动化

### 上午（2h）：核心概念理解

**什么是n8n？**
- 开源免费的可视化工作流自动化工具
- 支持**2000+**SaaS服务连接：Google、GitHub、钉钉、飞书、邮件...
- 适合做：Agent流水线自动化，比如"有新RSS → 总结 → 发消息给我"

**为什么需要工作流自动化？**
- 很多Agent任务其实就是"触发 → 处理 → 输出"
- 不用写代码，拖拖拽拽就能连起来
- 定时自动跑，解放双手

**核心概念：**
- **节点**：每个节点做一件事（触发、代码、API调用、LLM）
- **连线**：节点之间连起来，定义数据流向
- **表达式**：可以从上个节点取数据，传到下个节点

---

### 下午（2h）：动手搭建自动化流程

**目标：RSS订阅AI新闻 → LLM总结要点 → 自动推送到钉钉群**

**步骤1：Docker本地部署**
```bash
docker run -it --name n8n -p 5678:5678 -v ~/.n8n:/home/node/.n8n n8nio/n8n
```
启动后打开 `http://localhost:5678`，设置账号密码就能用。

**步骤2：创建工作流**

1. **添加触发节点：RSS Read**
   - 搜索"RSS"，选择"RSS Read"
   - 配置RSS地址：比如 https://www.feishu.cn/feed （换成你喜欢的AI新闻源）
   - 设置触发间隔（比如每24小时检查一次）

2. **添加LLM总结节点：OpenAI**
   - 搜索"OpenAI"，选择"OpenAI" → "Chat Completion"
   - 配置你的API Key
   - Prompt模板：
```
下面是一篇AI新闻，请用3句话总结核心要点，简洁明了：
{{ $json.allTitle }}
```
   - 从RSS节点取出新闻标题和内容，传给LLM

3. **添加输出节点：钉钉机器人**
   - 搜索"DingTalk"，选择"Send Message"
   - 配置你的钉钉机器人Webhook（钉钉群 → 群设置 → 智能群助手 → 添加机器人）
   - 消息内容填LLM总结的结果

4. **激活工作流**
   - 点击"Activate"，测试运行
   - 看新闻能不能正确总结并推送到钉钉

**成功了！** 现在每天自动给你推送总结好的AI新闻，不用你自己刷。

**练习扩展：** 试试改成推送到微信/邮件，或者加上情感分析。

## 🌐 第3天：Dify开源版本地部署

### 上午（2h）：Dify介绍

**什么是Dify？**
- **开源生产级**低代码Agent开发平台
- 比Coze更开放，可以私有部署，数据自己掌控
- RAG + Agent编排 + 发布一条龙
- 企业也能用，适合做内部知识库Agent

**核心特性：**
- ✅ 可视化Prompt编排
- ✅ 原生RAG支持，上传文档就能用
- ✅ 工具调用支持
- ✅ 一键发布API/Web/小程序
- ✅ 日志监控，对话日志可查

---

### 下午（2h）：Docker部署

**前提：** 需要你本地装了Docker和docker-compose。

**部署步骤：**

```bash
# 1. 克隆代码
git clone https://github.com/langgenius/dify.git

# 2. 进入docker目录
cd dify/docker

# 3. 复制环境配置文件
cp .env.example .env

# 4. 启动（需要拉取镜像，第一次几分钟）
docker-compose up -d
```

**检查启动：**
```bash
docker-compose ps
```
所有容器都是 `Up` 状态就成功了。

**访问：** 打开浏览器 `http://localhost`
- 第一次访问会让你设置管理员账号密码
- 进去后，左边菜单熟悉各个功能：
  - 应用：你的Agent应用列表
  - 知识库：文档管理
  - 工具：自定义工具配置
  - 设置：系统设置

**常见问题：**
- Q: 端口被占用了怎么办？
- A: 修改 `docker-compose.yml` 里的端口映射，改成其他端口

- Q: 拉镜像太慢？
- A: 配置Docker镜像加速，国内用阿里云镜像很快

## 🎯 第4天：在Dify搭建知识库Agent

### 上午（2h）：导入文档配置RAG

**步骤：**

1. **创建知识库**
   - 左侧"知识库" → "创建知识库"
   - 填写名称，选择Embedding模型（默认BGE就很好，中文）
   - 上传你的文档（支持PDF/Word/Markdown/TXT）

2. **处理配置**
   - 分段长度：默认500字符就不错
   - 重叠：默认50字符
   - 点击"下一步"开始处理，Dify自动分块+Embedding

3. **测试检索**
   - 处理完后，"检索测试" → 输入问题，看返回的片段相关不相关
   - 调整分段参数再测试，直到满意

---

### 下午（2h）：创建Agent应用

**步骤：**

1. **创建应用**
   - 首页"创建应用" → 选择"Agent"
   - 填写名称和描述

2. **编排Prompt**
   - Dify是可视化Prompt编排，拖拽就行
   - 系统提示词模板参考：
```
你是一个专业助手，基于用户的知识库回答问题。
如果知识库中找不到相关信息，请直接说"我在知识库中没有找到相关信息"，不要编造。
```

3. **关联知识库**
   - 点击"上下文" → 选择你刚才创建的知识库
   - 设置检索TopN（默认3-5就好）

4. **配置工具（可选）**
   - 如果你需要Agent能联网搜索，添加"搜索引擎"工具
   - 配置API Key即可

5. **测试发布**
   - 右侧测试框对话测试
   - 没问题后，点击"发布"，可以：
     - 发布成Web页面，分享给别人
     - 发布成API，集成到你的应用
     - 发布到飞书/钉钉/微信

**体验对比：**
- 和你手写LangGraph RAG对比
- 和Coze对比
- 各有什么优劣？记下来

## 🧩 第5天：Agent规划算法对比

### 上午（2h）：深入理解不同算法

现在Agent有多种规划算法，该怎么选？一张表看懂：

| 算法 | 全称 | 核心思想 | 工作方式 | 适用场景 | 推荐度 |
|------|------|----------|----------|----------|--------|
| **CoT** | Chain of Thought<br>思维链 | 让LLM一步一步思考，把推理过程说出来 | 单次调用，全程推理 | 简单问题，不需要外部工具，一步推理能搞定 | ⭐⭐⭐⭐ |
| **ToT** | Tree of Thought<br>思维树 | 把问题分解成思维树，探索多条路径，评估每个分支选最优 | 多步探索，选最好路径 | 复杂推理、创意写作、需要尝试多个方向 | ⭐⭐⭐ |
| **ReAct** | Reasoning + Acting<br>思考行动 | 思考 → 行动 → 观察循环，边做边想 | 循环调用，每次调用决定下一步做什么 | **大多数实用Agent，需要调用工具** | ⭐⭐⭐⭐⭐ |
| **Reflexion** | 自反馈 | 每一步做完自己反思，自我批评，下次改进 | 动作 → 反思 → 改进循环 | 需要自我改进的复杂任务 | ⭐⭐⭐ |

### 选择经验法则（直接用）：

1. **90%的商业Agent：选ReAct**
   - 需要工具调用，需要循环，ReAct最成熟

2. **简单问答推理：用CoT足够**
   - 不用循环，一次调用搞定，快又省token

3. **非常复杂的推理问题：才考虑ToT**
   - ToT探索多个路径，token消耗大，成本高
   - 只有确实需要才用

### 下午（2h）：动手对比体验

**任务：** 同一个问题，分别用三种方式问LLM，对比输出质量

比如问题：
> 小明有3个苹果，小红比小明多2个，小丽吃掉了2个苹果，其中1个是小明的，请问现在小明还有几个苹果？

分别这么问：

1. **CoT方式：**
```
请一步一步思考回答：
小明有3个苹果，小红比小明多2个，小丽吃掉了2个苹果，其中1个是小明的，请问现在小明还有几个苹果？
```

2. **ToT方式（简化版）：**
```
解决这个问题，有几种可能的思路，请分别列出每种思路的计算过程，然后给出你认为正确的答案：
小明有3个苹果...
```

3. **如果用ReAct（假设有计算器工具）：**
```
你可以调用计算器工具来帮助计算。
问题：...
```

**对比：**
- 哪种正确率最高？
- 哪种token消耗最少？
- 记录你的感受。

## 📏 第6天：Agent评估方法 + 复盘总结

### 上午（2h）：怎么科学评估你的Agent？

开发Agent，**先有评估，才有改进**。你得知道Agent好不好，才能知道往哪个方向优化。

五个核心评估维度：

### 1️⃣ 任务完成率（最重要！）

> **100个典型任务，Agent能成功完成多少个？百分比就是任务完成率**

这是硬指标，你的Agent好不好，就看这个。

**怎么构建测试集：**
- 收集用户真实问题，50-100个就够
- 每个问题标注正确答案
- 跑一遍，统计对不对，算完成率

**示例：**
- 你做了一个客服Agent，100个用户问题，75个回答正确 → 完成率75%

### 2️⃣ 工具调用正确率

对于带工具的Agent：
- 什么时候应该调用工具？Agent选对了吗？
- 参数填对了吗？
- 正确率 = 正确调用次数 ÷ 总调用次数

**常见错误：**
- 明明不需要调用工具，硬要调用
- 参数格式错了，执行失败
- 选错工具，用了搜索其实该用RAG

### 3️⃣ 回答准确性

对于问答/RAG Agent：
- 回答是不是正确？有没有幻觉？
- 来源对不对？有没有断章取义？

**量化方法：**
- 人工打分：1-5分
- LLM当裁判：用一个大模型给回答打分

### 4️⃣ 效率指标

- **平均步数**：完成任务平均走几步？步数越少越好
- **平均token消耗**：token = 钱，越少成本越低
- **平均耗时**：用户要等多久才能得到回答？

### 5️⃣ 用户体验

- 回答自然流畅吗？
- 流程符合用户预期吗？
- 用户会问了问题不知道Agent在干嘛吗？
